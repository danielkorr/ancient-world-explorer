Alexander Module Wireframes

Organization and map presentation options for adding Alexander the Great to VIA.

Recommendation

Preferred first step: internal module

Keep one deployed app and one shared map stack, but give Alexander its own mode/tab with campaign-specific controls.

currentWorld = "rome" | "alexander"

Keep extraction possible

Data already lives separately as js/alexander.js. If the module later needs its own brand, it can become a standalone app without rewriting the campaign model.

The current layer scaffold proves rendering and tap behavior. The next product move is to promote it from "checkbox layer" to "mode/tab".

Information Architecture

Option A: one app, separate modules

VIA
  • Roman World: roads, sites, quests, ORBIS, check-ins
  • Alexander: route, stops, phases, event panels
  • Future modules: Hannibal, Caesar, Persian Empire, Silk Roads

Option B: standalone Alexander app

Alexander Explorer
  • Own URL and product identity
  • Shared data conventions with VIA
  • External cross-links to VIA where places overlap

Data Organization

1. Phase Data

ALEXANDER_PHASES

  • Macedon
  • Asia Minor
  • Levant and Egypt
  • Persian Core
  • East and India
  • Return and Death

2. Stop Data

ALEXANDER_STOPS

  • event type
  • year and phase
  • lat/lng
  • certainty
  • narrative and significance

3. Route Data

ALEXANDER_ROUTES

  • from/to stop ids
  • generalized coords
  • certainty: solid/dashed/dotted
  • source note when reconstructed

The panel reads one selected stop. The map reads stops plus route segments. The phase control filters both.

Presentation Rules

Map

  • Route line is the main visual object.
  • Stops are secondary but tappable.
  • Dash pattern communicates uncertainty.

Panel

  • Event-first vocabulary.
  • Year, phase, type, certainty.
  • Source note visible when disputed.

Mode Switch

  • Rome keeps quests and ORBIS.
  • Alexander gets phases and timeline.
  • Shared basemap and panel shell.
