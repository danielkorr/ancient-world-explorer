## North star (the layer rethink)

A Codex review (`AGENTS.md`, 2026-06-14) re-examined VIA's data foundations. Its core
recommendation:

> **Itiner-e geometry + Pleiades identities + ORBIS travel modeling + Vici/Open Context
> evidence + modern OSM alignment.**

Itiner-e is the authoritative Roman-road dataset; ORBIS is a travel-time/cost *model*, not
road geometry. **VIA already reflects this** ‚Äî it ships Itiner-e as the road layer
(`js/roads-itinere.js`) and already uses ORBIS only for the travel-time card. The gap is
everything *downstream of geometry*: until now the Itiner-e layer carried **no metadata at
all** and was drawn as anonymous amber lines.

The most valuable opportunity Codex identified: Itiner-e's **certainty classification** maps
almost perfectly onto VIA's quest concept ‚Äî conjectured/hypothetical road stretches are
concrete, sourced field-verification opportunities, stronger than "missing Pleiades photo."

## Verified Itiner-e schema (v1.3 Zenodo dump, 14,769 features)

| Source field | Fill | Kept as | Notes |
|---|---|---|---|
| `Segment_s` | 100% | `cert` (`c`/`j`/`h`) | **certainty** ‚Äî Certain 7.0% / Conjectured 87.7% / Hypothetical 5.3% (by length: 2.73 / 89.82 / 7.45%) |
| `Name` | 100% | `name` | road name (e.g. "Tingi-Tamuda") |
| `Type` | 100% | `main: 1` | only when "Main Road" (Secondary omitted) |
| `Citation` | 100% | `cite` | contributor/editor |
| `Bibliograp` | 97% | `bib` | bibliography ref (e.g. "BAtlas 28") |
| `Itinerary` | 41% | `itin` | ancient-itinerary membership |

**Hard limits of the static dump** (do not over-promise): there is **no Pleiades-place
association field** and **no stable/persistent global segment id or URI** (only `InLine_FID`,
a non-unique per-input-line index). Those exist only in the Itiner-e *live API*. So this work
does not attempt offline segment‚ÜíPleiades joins or deep per-segment Itiner-e links.

## Step 1 ‚Äî what shipped on this branch

1. **Metadata capture** ‚Äî `scripts/build-roads.mjs` previously had a `KEEP_PROPS` allowlist of
   guessed field names that matched nothing, so every segment was bare `{coords}`. Replaced
   with the real fields above. Meta objects are **deduped** into a `ROADS_ITINERE_META` table
   (7,318 unique records) and each segment carries an integer index `m` into it. Regenerated
   `js/roads-itinere.js`: raw 1.65 ‚Üí 2.69 MB, **gzip ‚âà 0.55 MB** (delivery is gzipped).

2. **Certainty-legible styling** ‚Äî `js/app.js` styles each canvas polyline by `cert`. The user
   is red-green colorblind, so the **dash pattern is the primary channel**: Certain = solid,
   Conjectured = dashed, Hypothetical = dotted; hue/opacity only reinforce. Stays a quiet
   underlay beneath the curated named roads. Still on the shared `L.canvas()`.

3. **Tappable segments ‚Üí road panel** ‚Äî canvas polylines are not individual DOM nodes, so the
   existing `closest('path')` / divIcon touch delegation can't hit them (the documented roads
   landmine). Segments stay **non-interactive**; a single `map.on('click')` resolves the tap to
   the **nearest segment** (`findNearestItinere`, pixel distance with a zoom-sized bbox
   prefilter) and opens `showSegmentPanel`. Map-level click fires reliably on desktop *and*
   iOS (it already drove panel-close on mobile), so no new touch delegation was needed. The
   panel reuses `#info-panel`, showing name, certainty (plain-language), main/secondary,
   contributor, bibliography, itinerary membership, and an Itiner-e atlas link; site-only
   sections (ORBIS, check-in, quest progress, quest banner) are hidden via a panel "kind" gate.

4. **Cache-bust** ‚Äî `index.html` `?v=27 ‚Üí ?v=28`.

## Step 2 ‚Äî certainty as a quest (SHIPPED ‚Äî Option A)

**Decision (user): Option A.** Shipped: (1) a **Road Quest banner** in `showSegmentPanel` for
Conjectured/Hypothetical segments ("Verify this stretch" + a "Share this stretch ‚Üí" CTA via
`shareRoadSegment`/`onQuestCtaClick`; Certain segments get no banner); (2) a **roads certainty
filter** ‚Äî three `data-cert` rows in `#quest-legend` (`toggleCert`/`refreshVisibleRoads`) that
show only Certain / Conjectured / Hypothetical by mutating `itinereRoadsGroup` contents
(clearLayers + re-add subset, mobile-safe; rows inert while the Roads layer is off). Dash-pattern
swatches (`.legend-line` solid/dashed/dotted) keep the colorblind-safe channel in the legend.
Cache-bust v28 ‚Üí v29. The analysis that led here is kept below for the record.

Scoped against the real code so the decision has numbers behind it. The existing quest
system is **entirely site-marker + Pleiades-keyed**: `activeTiers` filters `allMarkers` by
`siteTier(site)` (`app.js:1223` `refreshVisibleMarkers`); `TOTAL_QUESTS` / `QUEST_PLEIADES`
(`app.js:1179`) drive progress + check-in credit off each site's `pleiades` id. Roads are
**line geometry with no Pleiades id and no markers** ‚Äî they live outside this system
entirely. That asymmetry is what separates the two options.

### Option A ‚Äî segment-panel CTA + roads certainty filter (RECOMMENDED)
- **Per-segment quest CTA:** in `showSegmentPanel`, add a "verify this stretch" call to
  action for Conjectured/Hypothetical segments (reuse the `#panel-quest-banner` DOM that's
  currently blanked for roads). This is the literal payoff of the certainty model ‚Äî those are
  field-verification opportunities.
- **Roads certainty filter:** a small roads-only control (3 rows with the solid/dashed/dotted
  swatches) that shows only Certain / Conjectured / Hypothetical. Implementation reuses the
  proven mobile-safe pattern ‚Äî keep `itinereRoadsGroup` permanently on the map, store each
  polyline ref on its `itinereSegs` entry, and `clearLayers()` + re-add the matching subset
  (never remove/re-add the group). New DOM + CSS for the control, parallel to the site legend.
- **Effort:** medium, fully self-contained, touches neither the site-quest filter nor
  check-ins. Honours "roads are a parallel axis to site quests."

### Option B ‚Äî first-class road-quest tier in the legend/quest system
- Folds roads into the site legend + `activeTiers` + quest-progress + check-ins.
- **The blocker:** that system is marker/Pleiades-based. To make road stretches real quest
  entries you'd either (i) spawn a representative marker per conjectured/hypothetical stretch
  ‚Äî **12,952 Conjectured + 777 Hypothetical** segments, which blows the ~1K divIcon budget and
  forces the **Phase 2 canvas-marker work first**; or (ii) extend the legend with a 4th axis
  that just filters the *roads layer* ‚Äî which is Option A wearing the legend's clothes. Plus
  check-in credit + progress assume a `pleiades` key roads don't have, so "road quests" need a
  **new contribution model** before they can count.
- **Effort:** large; collides head-on with the deferred canvas-rendering work and needs a road
  contribution model. Not the next step.

**Recommendation:** ship **Option A** next. It delivers the certainty‚Üíquest value (CTA +
filter) now, stays inside the static model, and doesn't preclude Option B's richer game loop
later ‚Äî once Phase 2 canvas markers and a road-contribution model exist.

## Deferred ‚Äî later steps (not Step 1 or 2)
- A certainty **filter / legend** UI on the roads layer.
- **Future phases (vision only, out of scope for the static model):** routable graph built from
  Itiner-e + ORBIS speeds; Vici/Open Context evidence attached to nearby segments; modern OSM
  alignment conflation; self-hosted PostGIS + vector tiles; a per-segment field-state
  classification (survives as road / path / buried / inaccessible / unverified). These need a
  backend/build and deliberately break VIA's pure-static model ‚Äî not now.
